1. 프로젝트 소개

aisum 에서 운영하는 다양한 웹·API 서비스(Django, FastAPI, Flask, PHP 등)는 결국 같은 구성 요소를 반복적으로 필요로 합니다 — nginx 리버스 프록시, SSL 인증서 발급·자동 갱신, 로그 영속화와 회전, 봇/스캐너 차단, Redis/Celery 의존성, 컨테이너 헬스체크와 자동 재시작. 이 작업은 그동안 프로젝트마다 미세하게 다른 설정 파일을 손으로 복제·수정하는 식으로 처리되어 왔고, 그 과정에서 작은 차이가 보안 구멍이나 운영 사고로 이어지는 경우가 적지 않았습니다.

aisum-infrakit 는 이 반복 작업을 단일 솔루션으로 묶어두기 위해 만든 사내 표준 인프라 템플릿입니다. 사내에서 가장 자주 쓰는 4가지 백엔드 패턴(Gunicorn WSGI / Uvicorn ASGI / uWSGI / PHP-FPM) 을 동일한 디렉토리·파일 패턴으로 미리 구성해 두었습니다. 새 프로젝트는 자신의 코드를 www/ 아래에 두고, 도메인 conf 생성기 한 번을 돌린 뒤 docker compose up -d 만 실행하면 production-ready 환경이 곧바로 동작합니다.

한눈에 보는 사용 흐름

git clone → www/ 에 앱 코드 배치 → nginx_http_conf.sh 로 도메인 conf 생성 → .env 작성 → docker compose up -d
누가무엇을 얻는가
새 서비스 개발자인프라 셋업에 며칠을 쓰지 않고 표준화된 production 환경에서 즉시 코드 실행에 집중. 도메인 발급 전이라도 Host: 헤더만으로 로컬 검증 가능.
DevOps / SRE사내 모든 서비스가 동일 구조라 트러블슈팅·보안 패치·인증서 갱신이 한 곳에서 일관되게 처리. 한 호스트에 여러 스택 공존 가능.
팀 리드권장 패턴(보안 헤더, CSP, 봇 차단, logrotate, host 헤더 인젝션 방어, catch-all 정책)을 코드로 박아 정책을 일원화 관리.

3. 아키텍처 & 4 스택

각 스택은 자체 docker-compose 디렉토리(compose/web-service/nginx_<stack>/)를 가지며, 독립적으로 띄우거나 한 호스트에 모두 동시 띄울 수 있습니다.

Client → HTTP/HTTPS nginx webserver
(80/443, SSL termination, 봇 차단) → proxy/uwsgi/fastcgi_pass app container
(gunicorn/uvicorn/uwsgi/php) ↔ redis (per stack)

Python 스택에서는 --profile celery 로 celery worker · celery-beat · flower(:5555) 세트가 추가되고, 이들은 redis 와 app 의 service_healthy 통과를 기다린 뒤 기동합니다.

5. docker compose 기반 동작

각 스택의 docker-compose.yml 은 webserver + app + redis 를 기본으로 정의하고, Python 스택은 --profile celery 로 celery 세트를, 모든 스택은 --profile redis 로 redis-stats 를 선택적으로 추가합니다.

서비스 구성과 의존성

서비스역할의존성 / 헬스체크
webservernginx 리버스 프록시. 80/443 listen, conf.d·nginx.conf·proxy_params·log·dhparam·letsencrypt 를 bind mount.depends_on: gunicorn-app (service_healthy) — app 준비 전엔 기동하지 않음
<stack>-appuv pip install --system 후 exec gunicorn .... PID 1 을 앱이 이어받아 docker stop 의 SIGTERM 을 직접 수신.bash TCP probe </dev/tcp/127.0.0.1/8000, interval 10s × retries 60 (최대 10분)

핵심 디자인 포인트 (docker-compose.yml 발췌)

docker-compose.yml  # app: 의존성 설치 → exec 로 PID1 승계 (SIGTERM 직접 수신)
  command: bash -c "uv pip install --system --no-cache -r requirements.txt \
                   && exec gunicorn -c /gunicorn/gunicorn.conf.py"

  # app: TCP probe healthcheck — 8000 accept 가능해지면 healthy
  healthcheck:
    test: ["CMD-SHELL", "bash -c '</dev/tcp/127.0.0.1/8000' 2>/dev/null"]
    interval: 10s ; timeout: 5s ; retries: 60 ; start_period: 30s

  # webserver: app 이 healthy 가 된 뒤에만 기동 → host not found in upstream race 회피
  depends_on:
    gunicorn-app:
      condition: service_healthy

  # redis: 평문 비밀번호를 redis.conf 에 두지 않고 --requirepass 로 주입
  command: ["sh", "-c", "exec redis-server /usr/local/etc/redis/redis.conf --requirepass \"$$REDIS_PASSWORD\""]

6. Dockerfile 구조

4개의 Dockerfile 이 docker/ 아래에 있습니다. 공통 원칙: ① apt 소스를 https(443)로 전환(아웃바운드 HTTP 80 차단 환경 대응 + 전송 구간 TLS 하드닝), ② cron + logrotate 패키지를 굽고 entrypoint 에서 cron 데몬 백그라운드 기동, ③ 운영 자동화(인증서 갱신·로그 회전·봇 리스트 갱신)를 이미지 안에 캡슐화.

7. config 설정 체계

nginx 설정은 전역(nginx.conf) → 봇 차단(ngxblocker) → 도메인 conf(conf.d) 의 계층으로 구성되고, 도메인 conf 는 생성기가 sample 템플릿에서 자동 생성합니다.

7.1 nginx.conf — 부트스트랩과 include 순서

nginx.conf · http{}# 순서가 중요 — 1) 해시 크기 선언 → 2) 수천 개 map 등록 → 3) 도메인 conf
include /etc/nginx/ngxblocker.d/botblocker-nginx-settings.conf;   # (1) server_names_hash_* 등
include /etc/nginx/ngxblocker.d/globalblacklist.conf;             # (2) $bad_bot / $bad_referer map
include /etc/nginx/conf.d/*.conf;                                 # (3) default.conf + 도메인 server 블록
⚠ 해시 directive 중복 금지

ngxblocker 가 server_names_hash_* / variables_hash_* 를 정의하므로, nginx.conf 에서 같은 directive 를 다시 정의하면 중복 정의 에러로 기동 실패합니다. 권장값은 ngxblocker 에 맡기고 건드리지 마세요.

7.2 봇 차단 (ngxblocker)

nginx-ultimate-bad-bot-blocker 의 두 영역:

영역위치 / context역할
ngxblocker.d/nginx.conf 의 http{} 가 includemap $http_user_agent $bad_bot {...} 등 수천 개 패턴 정의 (인스턴스 전체 공유)
bots.d/도메인 server{} 가 includeif ($bad_bot) { return 444; } — 매칭되면 응답 없이 TCP 연결 종료

차단 흐름: 요청 수신 → http map 이 $bad_bot 평가 → server 블록의 blockbots.conf 가 return 444. 444 는 nginx 전용 코드로 로그엔 남지만 클라이언트엔 응답이 가지 않아 봇 방어 효율이 높습니다. 갱신은 cron 이 6시간마다 GitHub raw 와 동기화하고 변경 시 nginx -t && nginx -s reload.

⚠ ngxblocker.d / bots.d 는 호스트 마운트가 아님

컨테이너 이미지 안에만 존재합니다. 화이트리스트 편집은 컨테이너 안에서 docker compose exec webserver vi /etc/nginx/bots.d/whitelist-domains.conf 로 하고, 영속이 필요하면 별도 volume 으로 분리하세요.

9. 로그 설정 & 관리 체계

로그 시스템은 ① 영속화(호스트 마운트), ② Docker 로그 드라이버 회전, ③ logrotate 엔진+정책 세 축으로 구성됩니다.

9.1 로그 영속화

  • 모든 nginx/app 로그는 호스트 log/<stack>/*.log 로 영속 마운트됩니다.
  • 파일명 패턴: <base>.<stack>_<type>_<protocol>.log — 예: example.com.gunicorn_access_http.log, example.com.gunicorn_error_https.log.
  • 컨테이너 stdout 은 Docker 로그 드라이버로도 캡쳐됩니다 (LOG_DRIVER=json-file, max-size=100m, max-file=5).

9.2 logrotate — 엔진과 정책의 분리

두 영역 중 하나라도 빠지면 회전이 동작하지 않습니다.

영역위치역할
엔진컨테이너 이미지 (docker/<stack>/Dockerfile)cron + logrotate 패키지 + aisum-logrotate.sh 헬퍼 + 매일 02:00 crontab. "언제·어떻게 회전을 트리거하는가".
정책 (dropin)호스트 script/logrotate/<stack>/* → /etc/logrotate.d/<name> 마운트"어떤 로그를 / 얼마 보관 / 어떤 postrotate". 회전 대상·retention·USR1 시그널.

엔진만 있고 정책 dropin 이 없으면 cron 은 매일 돌지만 /etc/logrotate.d/ 가 비어 어떤 로그도 회전되지 않습니다.

정책 dropin 의 기본 특징

각 dropin(예: script/logrotate/nginx/nginx)은 회전 정책을 선언적으로 정의합니다. 기본값은 다음과 같습니다:

  • 매일 점검 + 100M 초과 시 회전, 30개 보관 후 압축 보존(delaycompress).
  • nginx: 회전 후 USR1 시그널로 워커가 새 로그 fd 를 재오픈 (재시작 없음).
  • WSL/Windows 마운트 호환: dropin 에 su root root 가 명시되어 0777 마운트의 권한 검사를 우회합니다.
⚠ WSL bind mount "Potentially dangerous mode" 회피

Windows/WSL bind mount 는 parent 디렉토리를 0777 로 노출해 logrotate 가 회전을 거부합니다. 두 가지 회피책이 함께 적용되어 있습니다: (1) 컨테이너 entrypoint 가 /etc/logrotate.d/* 를 0644 로 sanitize 한 사본을 /run/logrotate.d/ 에 만든 뒤 회전. (2) 각 dropin 에 su root root 명시로 parent 권한 검사 우회.

HTTPS 를 처음 사용한다면 이 스크립트 실행이 필수입니다. 호스트의 script/ 가 webserver 컨테이너에 /script/ 로 마운트되어 있어 컨테이너 안에서 실행합니다. 대화형으로 도메인과 이메일을 입력받아 인증서 발급과 자동 갱신 등록까지 한 번에 처리합니다.

11.3 필수 패키지·dhparam 은 이미 이미지에 들어 있다

스크립트가 필요로 하는 패키지(certbot · cron · python3-certbot-nginx · ca-certificates · openssl)와 TLS 핸드셰이크용 dhparam.pem 은 docker/nginx/Dockerfile 빌드 단계에서 이미 이미지에 포함됩니다. 그래서 스크립트의 패키지 설치 단계는 인프라킷 이미지에서는 사실상 즉시 통과합니다.

dhparam 설계dhparam 생성은 CPU 를 오래 잡아먹으므로 컨테이너가 뜰 때마다 만들지 않습니다. 빌드 시 1회 생성해 이미지에 굽고, 첫 기동에서 호스트 ./ssl/dhparam/ 로 백업한 뒤, 재빌드·재기동 시 그 백업본을 자동 복원합니다 — 재생성 지연과 파라미터 교체를 동시에 막는 설계입니다. 따라서 letsencrypt.sh 는 dhparam 을 만들지 않고 인증서 발급에만 집중합니다. (이미지의 dhparam 은 전 도메인 공용이며 비밀이 아닙니다.)

11.4 이미 인증서가 있으면 재발급을 건너뛴다 — down 후에도 유지

스크립트는 발급 전에 test -d /etc/letsencrypt/live/<첫 도메인> 으로 판정합니다. certbot 은 발급에 성공했을 때만 이 디렉토리를 만들기 때문에, 존재 = "이미 유효한 인증서 있음" 의 신뢰할 수 있는 신호입니다. (불필요한 재발급으로 Let's Encrypt rate limit 을 소모하지 않기 위한 가드.)

docker compose down 후에도 재발급이 생략된다

발급 인증서 경로 /etc/letsencrypt 는 호스트 ./ssl/letsencrypt/ 로 bind mount 되어 있습니다. docker compose down 은 컨테이너를 지우지만 호스트 폴더는 그대로 두므로(bind mount 는 -v 로도 삭제되지 않음), down/up 으로 컨테이너가 초기화돼도 live/<domain> 이 살아 있어 다음 실행에서 재발급을 건너뜁니다. 인증서를 정말로 새로 받으려면 호스트의 ssl/letsencrypt/ 내용을 직접 비우세요.

11.5 www/certbot 의 역할과 cron reload

www/certbot — ACME 챌린지 웹루트. Let's Encrypt 는 http://<도메인>/.well-known/acme-challenge/ 로 접속해 도메인 소유를 검증합니다. certbot 이 토큰 파일을 /www/certbot 아래 쓰고 nginx 가 그 경로를 서빙하므로, certbot 의 -w /www/certbot 와 도메인 conf 의 root /www/certbot 가 일치해야 발급이 성공합니다(sample conf 에 이미 포함, 앱 코드와 분리된 전 도메인 공용 폴더).

cron reload — 갱신을 무중단으로 반영. 인증서는 약 90일이면 만료됩니다. cron 이 매주 갱신을 점검하고, 갱신이 실제로 일어나면 --deploy-hook 이 nginx -t && nginx -s reload 를 실행합니다. 갱신은 디스크의 인증서 파일만 바꾸므로 실행 중인 nginx 가 새 인증서를 다시 읽도록 reload 가 필요하며, 이 reload 는 요청이 끊기지 않는 graceful 동작입니다. (컨테이너엔 systemd 가 없어 cron + nginx -s reload 를 씁니다.)

🚧 전제 조건 (반드시 충족)
  • nginx 가 먼저 HTTP(:80) 로 떠 있어야 함 — ACME 챌린지에 응답해야 발급됩니다.
  • 도메인 A 레코드가 호스트를 가리키고 80 포트가 외부 도달 가능해야 함 (방화벽/Cloudflare 가 80 을 막으면 실패).
  • 호출 순서: HTTP conf → compose up → letsencrypt.sh → HTTPS conf → reload.

12. HTTPS 전환 절차

발급 자체가 ACME webroot challenge 라서 nginx 가 먼저 HTTP 로 띄워져 있어야 합니다. 따라서 순서가 고정되어 있습니다:

nginx_http_conf.sh→ compose up -d→ letsencrypt.sh→ nginx_https_conf.sh→ nginx -s reload

전체 절차 (공인 도메인)

  1. HTTP 도메인 conf 먼저 생성 — ACME 챌린지 webroot 제공

    cd config/web-server/nginx/gunicorn
./nginx_http_conf.sh -w django_sample -p 80 -d example.com -a gunicorn-app -s 8000 -n example
    왜 HTTP 먼저인가Let's Encrypt 발급은 http://example.com/.well-known/acme-challenge/ 를 읽어 도메인 통제를 검증합니다. 즉 인증서를 받기 전에 이미 :80 으로 응답할 수 있어야 합니다. 이 단계에서 만든 HTTP conf 에는 sample 템플릿이 제공하는 ACME 챌린지 location (root /www/certbot)이 들어 있어, 다음 단계의 발급이 이 경로로 검증됩니다. HTTPS conf 를 먼저 만들면 아직 없는 인증서 파일을 참조해 nginx -t 가 실패하므로 순서가 고정입니다.

  2. nginx 기동 (이 시점엔 HTTP 만 listen)

    cd ../../../../compose/web-service/nginx_gunicorn
docker compose up -d --build
    이 단계의 상태컨테이너가 떠서 :80 으로 요청을 받기 시작합니다. 아직 443 listen 도, 인증서도 없습니다 — 목적은 오직 ACME 챌린지에 응답할 수 있는 살아있는 nginx 를 띄우는 것입니다. curl -H "Host: example.com" http://localhost/ 로 백엔드까지 정상인지 미리 확인해 두면 이후 단계의 변수를 줄일 수 있습니다.

  3. 인증서 초기 발급 — 컨테이너 안에서 letsencrypt.sh 실행

    docker compose exec webserver /script/letsencrypt.sh
#   Enter the service domain  > example.com www.example.com
#   Enter the user e-mail     > you@example.com
    여기서 일어나는 일스크립트가 도메인·이메일을 입력받아 certbot certonly --webroot -w /www/certbot 으로 발급을 시도하고, 성공하면 /etc/letsencrypt/live/example.com/ 에 fullchain.pem · privkey.pem · chain.pem 이 생깁니다. 이 경로는 호스트 ./ssl/letsencrypt/ 로 영속되므로 이후 down/up 에도 살아남고, 재실행 시 재발급을 건너뜁니다(11.4). 동시에 주간 자동 갱신 cron 도 등록됩니다. 자세한 동작은 11. letsencrypt.sh 동작 참조.

  4. HTTPS 도메인 conf 생성 (이제 cert 가 존재 → nginx -t 통과)

    cd ../../config/web-server/nginx/gunicorn
./nginx_https_conf.sh -w django_sample -p 80 -d example.com -a gunicorn-app -s 8000 -n example
# → conf.d/example_gunicorn_ng_https.conf
    이제 안전한 이유HTTPS conf 는 /etc/letsencrypt/live/example.com/fullchain.pem 같은 인증서 경로를 직접 참조합니다. 3단계에서 그 파일들이 실제로 생겼기 때문에, 지금 만든 conf 는 nginx -t 검증을 통과합니다. 생성된 파일에는 80→443 리다이렉트 server 와 443 메인 server 가 함께 들어 있어, 적용 후에는 평문 접속이 자동으로 HTTPS 로 넘어갑니다.

  5. HTTP conf 정리 → reload → 검증

    rm conf.d/example_gunicorn_ng_http.conf       # ⚠ 같은 도메인 HTTP/HTTPS 공존 금지 (아래 주의)
cd ../../../../compose/web-service/nginx_gunicorn
docker compose exec webserver nginx -t
docker compose exec webserver nginx -s reload
curl -I https://example.com/
    마지막 정리가 중요1단계의 HTTP conf 와 4단계의 HTTPS conf 는 같은 upstream example_backend 이름을 정의합니다. 둘이 conf.d/ 에 공존하면 duplicate upstream 으로 nginx 기동이 실패하므로, HTTP conf 를 삭제합니다(HTTPS conf 가 이미 80→443 리다이렉트를 포함하므로 평문 접속도 계속 처리됨). 이후 nginx -t → nginx -s reload 로 무중단 적용하고, curl -I https://example.com/ 로 200 과 Strict-Transport-Security 헤더를 확인합니다.
⚠ 가장 흔한 함정 — duplicate upstream

같은 도메인의 HTTP conf 와 HTTPS conf 가 conf.d/ 에 동시에 존재하면 두 파일이 같은 upstream <name>_backend 이름을 정의해 nginx: [emerg] duplicate upstream "<name>_backend" 로 기동 실패합니다. HTTPS 전환이 끝나면 HTTP conf 는 제거하거나 80→443 redirect 만 남기세요. (HTTPS 템플릿에는 이미 80→443 redirect server 블록이 포함되어 있으므로 HTTP conf 는 삭제하는 것이 깔끔합니다.)

HTTP conf 와 HTTPS conf 는 무엇이 다른가

둘 다 같은 백엔드로 프록시하는 점은 같지만, HTTPS conf 는 암호화 계층과 그에 따른 보안 설정이 추가됩니다. 쉽게 말해 "HTTP conf + (TLS 종료 + 평문 차단 + HTTPS 전용 보안 헤더)" 가 HTTPS conf 입니다.

항목HTTP confHTTPS conf
listen 포트80 한 개443(ssl, http2) + 80(리다이렉트 전용)
평문 80 요청그대로 서비스301 로 HTTPS 로 강제 전환
인증서없음fullchain.pem·privkey.pem(certbot) + dhparam.pem(이미지)
TLS 옵션없음TLSv1.2/1.3, 세션 캐시, OCSP 스테이플링 등
보안 헤더CSP·X-Content-Type-Options 등위 + HSTS(HTTPS 강제, HTTP 에선 무의미하므로 제외됨)
봇 차단·정적 서빙·백엔드 프록시동일
한 줄 요약HTTP conf 는 "받아서 백엔드로 넘긴다"가 전부이고, HTTPS conf 는 그 앞에 "안전하게 받아서(TLS) · 평문은 막고(301·HSTS) · 넘긴다" 가 추가된 형태입니다. 그래서 HTTPS 전환은 동작을 바꾸는 게 아니라 같은 서비스에 보안 계층을 덧씌우는 작업이며, 백엔드(upstream)·정적 파일·봇 차단 규칙은 그대로 재사용됩니다.

로컬 self-signed (도메인·외부 도달 없이 HTTPS 흐름만 확인)

self-signed 가 무엇인가"자체 서명(self-signed)" 인증서는 Let's Encrypt 같은 공인 CA(인증기관)의 서명 없이, 자기 자신이 자신을 보증하도록 openssl 로 직접 만든 인증서입니다. 암호화(TLS 핸드셰이크·세션 키 교환)는 공인 인증서와 똑같이 동작하지만, "이 서버가 정말 그 도메인의 주인인가"를 검증해 줄 제3자가 없어 신뢰 체인이 없습니다. 그래서 브라우저는 "이 연결은 안전하지 않습니다" 경고를 띄웁니다.
언제 쓰는가 — 순수 테스트 목적이 절차는 공인 도메인도, 80/443 외부 도달도 없는 환경에서 HTTPS 설정 자체가 올바른지(인증서 경로·SSL 옵션·443 listen·리다이렉트)를 확인하기 위한 테스트 용도입니다. Let's Encrypt 발급은 외부에서 도메인에 접속할 수 있어야 가능하지만, self-signed 는 인터넷 연결이나 실제 도메인 없이 로컬에서 즉시 만들 수 있어 개발·CI 검증에 적합합니다. 도메인 통제 검증(ACME)을 거치지 않으므로 letsencrypt.sh 도 실행하지 않습니다. 
./nginx_https_conf.sh -w django_sample -p 80 -d localhost -a gunicorn-app -s 8000 -n localhost
docker compose up -d
# 컨테이너 안에서 conf.d 가 참조하는 모든 cert 경로에 self-signed 인증서를 생성
docker compose exec webserver sh -c '... openssl req -x509 -nodes -newkey rsa:2048 ... ; nginx -t && nginx -s reload'
curl -k --resolve localhost:443:127.0.0.1 https://localhost/    # -k = 인증서 검증 생략
⛔ 운영 사용 금지

self-signed 인증서는 신뢰 체인이 없어 브라우저 경고가 뜨고, 사용자가 경고를 무시하도록 길들이면 중간자 공격(MITM)에 취약해집니다. 실제 서비스에는 절대 사용하지 말고, 공인 도메인이 준비되면 위 전체 절차(공인 도메인) 로 Let's Encrypt 인증서를 발급해 교체하세요. curl -k 의 -k 도 검증을 생략하는 옵션이므로 테스트에서만 사용합니다.

자동 갱신

letsencrypt.sh 가 등록한 cron 이 매주 월 05:00 certbot renew --quiet --deploy-hook "nginx -t && nginx -s reload" 를 실행합니다. 운영 중 재호출 불필요(도메인을 새로 추가할 때만 재실행). 수동 점검:

docker compose exec webserver certbot renew --dry-run
docker compose exec webserver certbot certificates        # 발급 목록 / 만료일
tail -100 log/nginx/crontab_*.log                         # 최근 cron 결과

13. 주의사항 총정리

아래는 실제로 가장 자주 사고로 이어지는 항목입니다. 치명 은 기동 실패·보안 노출로 직결되고, 정합성 은 값이 어긋나면 조용히 오동작하며, 운영 은 알아두면 시간을 아끼는 항목입니다.

구분주의사항 · 왜 / 어기면 무슨 일이
순서HTTPS 는 HTTP conf → compose up → letsencrypt.sh → HTTPS conf → reload 순서를 지킬 것. HTTPS conf 는 fullchain.pem·privkey.pem 경로를 직접 참조하는데, 발급(letsencrypt.sh) 전에 HTTPS conf 를 올리면 그 파일이 없어 nginx -t 가 cannot load certificate 로 실패하고 컨테이너가 기동 루프에 빠집니다. 발급 자체도 :80 의 ACME 응답이 먼저 떠 있어야 가능하므로 순서가 물리적으로 강제됩니다.
충돌같은 도메인의 HTTP conf 와 HTTPS conf 를 conf.d/ 에 동시에 두지 말 것. 두 파일 모두 upstream <name>_backend 라는 같은 이름의 블록을 정의하기 때문에 nginx 가 duplicate upstream "<name>_backend" emerg 로 기동을 거부합니다. HTTPS 전환이 끝나면 HTTP conf 는 삭제하세요 — HTTPS conf 에 이미 80→443 리다이렉트가 포함되어 평문 접속도 계속 처리됩니다.
보안default_server 는 포트(80/443)당 정확히 1개, default.conf 에만 둘 것. 도메인 conf 에 실수로 default_server 를 붙이면 어느 server 블록이 catch-all 이 되는지가 비결정적이 되어, 알 수 없는 Host/SNI 요청이 운영 도메인 블록으로 흘러들어 그 도메인의 인증서·정적 자원이 노출될 수 있습니다. default.conf 는 그런 요청을 return 444/ssl_reject_handshake 로 즉시 끊는 보안 장치입니다.
정합성REDIS_PASSWORD 와 CELERY_BROKER_URL 의 password 부분을 같은 값으로 맞출 것. redis 는 --requirepass 로 그 비밀번호를 요구하는데 celery 가 다른 값으로 접속하면 NOAUTH/연결 거부로 celery 가 무한 재시작합니다. URL 에 들어가므로 @ : / 같은 reserved char 는 피하거나 URL-encode 하고, 외부 노출되는 flower 의 FLOWER_PWD 기본값도 반드시 교체하세요.
정합성생성기 -w = .env 의 PROJECT_DIR = www/ 의 실제 폴더명, 세 값이 일치해야 함. 어긋나면 app 컨테이너는 빈/엉뚱한 디렉토리에서 기동하고(혹은 requirements.txt 를 못 찾아 죽고), nginx 의 /static·/media alias 가 존재하지 않는 경로를 가리켜 404 가 납니다.
정합성certbot 의 -w /www/certbot 와 도메인 conf 의 ACME root /www/certbot 가 일치해야 함. 둘이 다르면 certbot 이 토큰을 쓴 위치와 nginx 가 서빙하는 위치가 어긋나 Let's Encrypt 가 챌린지 파일을 찾지 못하고 발급이 실패합니다. sample 템플릿은 이미 /www/certbot 로 고정되어 있으니 임의로 앱 루트로 바꾸지 마세요.
중복nginx.conf 에서 server_names_hash_* / variables_hash_* 를 직접 정의하지 말 것. 이 값들은 ngxblocker 의 botblocker-nginx-settings.conf 가 이미 선언합니다(수천 개 map 을 수용하려고 크게 잡아둠). nginx.conf 에서 같은 directive 를 또 쓰면 중복 정의 에러로 기동에 실패합니다.
includeproxy_params/uwsgi_params/fastcgi_params 는 반드시 location 안에서만 include. http{} 전역에 두면 정적 파일 서빙·리다이렉트 등 모든 컨텍스트에 헤더 세팅이 강제 적용되어, 의도치 않은 헤더 변형이나 백엔드 정보 노출이 발생합니다.
rate limitlimit_req_zone/limit_conn_zone 은 nginx.conf 와 도메인 conf 양쪽 주석을 동시에 해제할 것. zone 정의(nginx.conf)와 사용(도메인 conf) 중 한쪽만 켜면 unknown limit_req zone "general" 로 기동이 실패합니다. 또 zone=addr 이름은 ngxblocker 가 이미 쓰므로 직접 재정의하지 말고 그 zone 을 참조하세요.
로그clone 직후 log/ 디렉토리가 있는지 확인. 없으면 nginx 가 open() "/log/nginx/error.log" failed 로 기동 즉시 종료합니다. mkdir -p log/{nginx,gunicorn,uvicorn,uwsgi,php-fpm} 로 미리 만들어 두세요(.gitkeep 으로 추적되지만 누락될 수 있음).
멀티스택호스트당 80/443 외부 노출은 1 스택만. 여러 스택을 동시에 외부로 띄우면 포트가 충돌하므로, 일부 스택의 ports: 를 8080:80 처럼 바꾸거나 앞단에 별도 reverse proxy(Cloudflare/nginx/traefik)를 두세요. flower 5555 도 여러 스택에서 --profile celery 를 켜면 충돌하니 한 스택에서만 활성화합니다.
인증서 보존인증서는 docker compose down 후에도 남습니다. ./ssl/letsencrypt/ 호스트 마운트 덕분에 컨테이너를 내려도 발급 인증서가 보존되어 재발급이 생략됩니다(11.4). 반대로, 인증서를 정말로 새로 받고 싶다면 컨테이너만 내릴 게 아니라 호스트의 ssl/letsencrypt/ 내용을 직접 비워야 합니다.
덮어쓰기conf 생성기는 같은 이름 파일을 항상 덮어쓰고, nginx 를 자동 reload 하지 않습니다. 수정 중인 conf 가 있으면 미리 백업하고(백업은 호출 측 책임), 생성 후에는 안내되는 nginx -t && nginx -s reload 를 직접 실행해야 변경이 반영됩니다.

14. 트러블슈팅
